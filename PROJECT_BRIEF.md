## SCOPE
- **영역**: 4개 전부 (희앤썬 그룹 / AC 호텔·팰리스 / AC Kitchen / KLOUD)

## 운영 원칙
1. **최소 예산 · 인하우스 + 파트너십 중심**
2. **모든 액션은 4요소를 갖춘다**: 누가(R&R) / 언제(주 단위) / 얼마(인하우스 vs 파트너 비용 구조) / 지표(측정 가능 KPI)
3. **일반론 금지**: "SNS를 활용한다", "인지도를 높인다" 같은 문장은 거부 사유. 항상 채널·콘텐츠·KPI까지 분해.
4. **스코프 경계 준수**: 한 영역을 수정할 때 다른 영역에 영향이 가는지 명시. 변경하지 않는 부분은 명시적으로 "유지".
5. **시각화는 즉시 렌더링 가능한 형태**: 마크다운 표 또는 Mermaid 코드 블록.

## ★ DECK V2 작업 방식 (이 프로젝트 전용)

### 자산 인벤토리 (필수 컨텍스트)
프로젝트 Knowledge에 다음 파일들이 있다. 작업 전 먼저 확인:
- **22개 슬라이드 HTML** (`slide-XX-*.html`) — 편집 대상
- **README.md** — 작업 가이드
- **DESIGN_TOKENS.md** — 공통 CSS / 폰트 / 패턴
- **SLIDE_INDEX.md** — 22 슬라이드 카피·자산·결합 인덱스

### 작업 흐름 (필수 준수)
1. 사용자 요청 분석 → 어느 슬라이드(들)를 건드리는지 명시
2. 변경 spec 제안 — 표로 (변경 전 / 변경 후 / 영향)
3. **사용자 합의** 받은 후에만 수정 시작
4. `view`로 해당 HTML 확인 → `str_replace`로 정확한 부분만 변경
5. `present_files`로 결과 표시
6. 변경 영향이 다른 슬라이드에 가면 "유지" 선언 명시

### 파일 위치 규약
- 슬라이드 HTML: 사용자가 업로드한 위치 또는 `/mnt/user-data/uploads/`
- 수정한 결과물: `/mnt/user-data/outputs/`
- 최종 export (PPT/통합 HTML): `/mnt/user-data/outputs/`

### 변경 시 점검 사항
- DESIGN_TOKENS.md에 정의된 색상·폰트 변수 위배 여부
- 다른 슬라이드와의 카피 ring (예: Slide 1 _More Than Hotels._ ↔ Slide 21 _Lifestyle._) 일관성
- FIXED FACTS 위배 여부

### Export 명령
사용자가 "deck 다시 빌드해줘" / "PPT/HTML 다시 만들어줘" 하면:
1. `scripts/render_pngs.py` 실행 — 22 HTML → PNG
2. `scripts/build_pptx.py` 실행 — PNG → PowerPoint
3. `scripts/build_deck_html.py` 실행 — PNG → 단일 HTML
4. `present_files`로 두 파일 출력
